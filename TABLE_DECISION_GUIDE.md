# 太初命理系统 - 数据库表决策指南

> 本文档帮助您理解每个表的具体功能、是否应该保留、以及如何使用

---

## 表分类决策矩阵

### 第一类：✅ 必须保留 - 核心业务表

这些表直接支持系统的主要功能，不能删除。

#### 用户系统
```
┌─────────────────────────────────────────────────────────┐
│ users                          【用户账户主表】         │
├─────────────────────────────────────────────────────────┤
│ ✅ 必须                                                 │
│ 💾 用途：存储所有用户账户信息                           │
│ 📊 规模：应用启动后不断增长                             │
│ 🔑 关键字段：user_id, phone, nickname, points,         │
│             vip_level, created_at                       │
│ 📈 增长速度：中等（每新用户+1行）                       │
│ ❓使用场景：登录、用户信息查询、个人中心展示           │
└─────────────────────────────────────────────────────────┘
```

#### 积分系统
```
┌─────────────────────────────────────────────────────────┐
│ points_records                 【积分交易记录】         │
├─────────────────────────────────────────────────────────┤
│ ✅ 必须                                                 │
│ 💾 用途：记录每一笔积分变动（加/减）                   │
│ 📊 规模：高频数据，需要定期归档                         │
│ 🔑 关键字段：user_id, type (add/reduce), points,       │
│             action, created_at                          │
│ 📈 增长速度：快（用户每操作+1行）                       │
│ ❓使用场景：                                             │
│   • 积分充值/消耗记录                                   │
│   • 积分流水账查询                                      │
│   • 用户积分来源追踪                                    │
│   • 后台积分统计与核对                                  │
│ ⚠️  建议：每月/每季度归档老数据到历史表                │
└─────────────────────────────────────────────────────────┘
```

#### 支付系统
```
┌─────────────────────────────────────────────────────────┐
│ recharge_orders                【积分充值订单】         │
├─────────────────────────────────────────────────────────┤
│ ✅ 必须                                                 │
│ 💾 用途：记录所有充值订单，对应支付平台的订单号         │
│ 📊 规模：核心财务数据，必须精确保存                     │
│ 🔑 关键字段：order_no, user_id, amount, status,        │
│             payment_type, pay_order_no, created_at     │
│ 📈 增长速度：中等                                       │
│ ❓使用场景：                                             │
│   • 订单查询与支付状态追踪                              │
│   • 财务对账                                            │
│   • 支付异常处理                                        │
│ ⚠️  建议：永不删除，财务审计需要完整历史               │
└─────────────────────────────────────────────────────────┘
```

#### 命理记录

├─────────────────────────────────────────────────────────┤
│ ✅ 必须                                                 │

## 💬 常见问题

### Q1: 这些表对系统运行都必须吗？
**A:** 不是的。核心业务表（users, points_records等）必须，但SEO表、某些扩展表是可选的。

### Q2: 可以删除历史数据来节省空间吗？
**A:** 
- **财务数据**（recharge_orders）：永远不要删除
- **用户操作记录**（operation_logs）：可以按年份归档
- **积分记录**：保留最近1年的，老数据归档
- **访问日志**：可以定期清理

### Q3: 如何提高表的查询性能？
**A:**
1. **建立适当的索引**（已有的外键字段、时间字段）
2. **定期优化表**：`OPTIMIZE TABLE table_name;`
3. **考虑缓存**（Redis）：常查的配置、用户信息
4. **分库分表**：如果表超过1000万行

### Q4: 如何备份这些表？
**A:**
```bash
# 完整备份
mysqldump -u user -p taichu > backup_$(date +%Y%m%d).sql

# 只备份结构
mysqldump -u user -p --no-data taichu > structure_$(date +%Y%m%d).sql

# 只备份关键表
mysqldump -u user -p taichu users recharge_orders > critical_$(date +%Y%m%d).sql
```

### Q5: 表的字符集应该如何设置？
**A:** 所有表都应该使用 `utf8mb4 COLLATE utf8mb4_unicode_ci`
- utf8mb4：支持表情符号和所有Unicode字符
- unicode_ci：不区分大小写的排序
