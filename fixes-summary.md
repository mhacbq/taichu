# 太初命理网站问题修复总结

## 修复完成时间
2025年3月15日

## 修复内容概览

### 1. 合婚导出功能 - 实现真正的HTML报告生成 ✅
**文件**: `backend/app/controller/Hehun.php`

**问题**: 合婚导出功能返回假的URL，用户支付积分后无法获得实际报告

**修复**: 
- 实现完整的HTML报告生成逻辑
- 使用美观的CSS样式设计报告页面
- 报告包含五维度评分、专业建议、流年分析、化解方案
- 支持打印为PDF格式
- 报告文件保存在服务器上，提供真实可访问的URL

### 2. 订单号生成重复风险 - 修复 ✅
**文件**: `backend/app/model/RechargeOrder.php`

**问题**: 使用 `mt_rand(10, 99)` 在高并发下可能重复

**修复**:
- 使用微秒级时间戳代替秒级时间戳
- 使用 `random_int()` 生成4位随机数（比 `mt_rand` 更安全）
- 添加进程ID作为唯一标识
- 订单号格式：`YmdHis` + `微秒(6位)` + `随机数(4位)` + `进程ID(4位)`

### 3. 支付金额浮点精度问题 - 修复 ✅
**文件**: `backend/app/controller/Payment.php`

**问题**: `(int) (0.1 * 100)` 在某些PHP版本等于 `9` 而非 `10`

**修复**:
- 使用 `round()` 函数代替直接强制转换
- 代码: `$expectedFee = (int) round($order->amount * 100);`

### 4. 分享码无限递归风险 - 修复 ✅
**文件**: `backend/app/model/TarotRecord.php`, `backend/app/model/BaziRecord.php`

**问题**: 分享码用尽时可能导致栈溢出

**修复**:
- 添加最大重试次数限制（默认10次）
- 超过最大重试次数后使用 `uniqid()` 生成唯一码
- 避免递归调用，改用循环

### 5. Tarot积分扣除事务嵌套问题 - 修复 ✅
**文件**: `backend/app/controller/Tarot.php`

**问题**: 控制器开启事务后调用模型方法可能嵌套事务

**修复**:
- 使用统一的数据库连接 `Db::name()` 代替模型
- 添加行锁 `lock(true)` 确保并发安全
- 在事务内重新查询用户积分，确保数据最新
- 优化错误处理和日志记录

### 6. API响应敏感数据过滤 - 修复 ✅
**文件**: `backend/app/model/BaziRecord.php`

**问题**: API返回了 user_id 等敏感字段

**修复**:
- `getUserHistory()` 方法添加 `field()` 限制返回字段
- `getUserHistoryPaged()` 方法添加 `field()` 限制返回字段
- 排除敏感字段如 `user_id`

### 7. 首次排盘免费逻辑不完善 - 修复 ✅
**文件**: `backend/app/controller/Paipan.php`

**问题**: 用户删除历史记录后可重复获取免费排盘

**修复**:
- 双重检查：同时检查 `BaziRecord` 和 `PointsRecord`
- 即使删除排盘记录，积分消费记录仍然存在
- 有效防止绕过免费限制

### 8. 前端积分检查竞态条件 - 修复 ✅
**文件**: `frontend/src/views/Tarot.vue`

**问题**: 页面显示的积分余额可能过期，导致用户误以为积分充足

**修复**:
- 在 `showConfirm()` 方法中重新获取最新积分
- 在确认对话框显示前进行最终积分检查
- 避免用户在不同页面消费积分后的状态不一致问题

### 9. RateLimit区分VIP用户 - 修复 ✅
**文件**: `backend/app/middleware/RateLimit.php`

**问题**: VIP用户和普通用户使用相同的限流限制

**修复**:
- 添加 `isVipUser()` 方法检测用户VIP状态
- VIP用户限流限制放宽2倍
- 添加响应头 `X-RateLimit-VIP` 标识VIP状态
- VIP状态缓存5分钟，减少数据库查询

### 10. 日志记录优化 - 检查 ✅

**结论**: 现有日志记录合理，主要是支付回调相关的重要日志，无需调整。
